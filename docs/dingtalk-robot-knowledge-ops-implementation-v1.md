# 钉钉机器人知识运营后台实施文档（V1）

## 1. 文档目的

本文档用于指导“钉钉机器人知识运营后台（V1）”从方案阶段进入实施阶段，明确：

- 实施目标
- 实施范围
- 实施顺序
- 前后端分工
- 联调与验收要求
- 进度记录方式

---

## 2. 实施目标

在不改变“员工通过钉钉单聊机器人提问”这一主入口的前提下，交付一个面向业务同事的后台，用于：

- 录入与维护机器人知识
- 进行钉钉对话验证
- 将知识发布到机器人
- 按角色控制维护权限

V1 的核心闭环是：

```text
录入知识
→ 钉钉对话验证
→ 发布到机器人
```

只要这条链路打通，V1 即具备业务可用性。

---

## 3. 实施范围（V1）

### 3.1 知识类型
V1 固定支持 4 类知识：

1. 制度文档
2. FAQ
3. 固定报价
4. 受控文档

### 3.2 页面范围
V1 后台页面范围：

1. 机器人首页
2. 我的待办
3. 机器人知识管理
4. 上传与导入
5. 待确认内容
6. 钉钉对话验证
7. 发布到机器人
8. 权限与角色

### 3.3 角色范围
V1 角色边界固定为：

- 人事：员工手册 / FAQ / 受控文档
- 商务：固定报价
- 财务：只读验证
- 管理员：全权限

---

## 4. 非目标（V1 不做）

V1 暂不纳入：

- 员工侧独立 Web 前台
- 自动全量知识抽取与自动发布
- 复杂审核流
- 高级版本回滚
- 复杂运营分析大屏
- 流程指引配置（`flow_guidance`）后台化

---

## 5. 实施原则

### 5.1 钉钉优先
后台不是独立知识网站，而是**钉钉机器人运营后台**。

### 5.2 主链路优先
优先打通：
- FAQ 录入
- 固定报价录入
- 钉钉验证
- 发布到机器人

### 5.3 后台与运行时接口分离
后台接口不要混入现有钉钉消息事件处理接口。

### 5.4 权限前置
角色权限必须在页面显隐和接口校验两层都落地。

### 5.5 发布前必须验证
所有知识在发布前必须支持钉钉对话验证。

目标：把后端主链路打通。

### 内容

### 阶段产出

- 基础数据结构稳定

---

## 阶段 B：前端主链路页面
目标：打通录入 → 验证 → 发布的最小前台操作闭环。

### 内容
- 后台框架页
- 机器人知识管理页
- FAQ 录入页
- 固定报价录入页
- 制度文档上传流程页
- 受控文档录入页
- 钉钉对话验证页
- 发布到机器人页

### 阶段产出
- 业务同事可以完成一次完整知识发布流程

---

## 阶段 C：运营辅助页
目标：补齐更适合日常使用的运营能力。

### 内容
- 机器人首页
- 我的待办
- 待确认内容页
- 批量导入能力
- 权限与角色页（管理员）

### 阶段产出
- 后台具备日常运营条件

---

## 阶段 D：联调与验收
目标：验证后台与钉钉机器人行为一致。

### 内容
- 前后端联调
- 权限联调
- 验证结果与机器人实际行为校验
- 发布链路验收

### 阶段产出
- V1 验收完成

## 10. 风险点与应对

## 风险 1：文档越写越多，迟迟不开发
### 应对
从现在开始，优先转开发，不再扩展大方案文档。

## 风险 2：权限边界在开发中被打散
### 应对
严格以已锁定 V1 规则为准，不临时扩权。

## 风险 3：前后端接口命名不统一
### 应对
统一采用已定义的 API schema 文档。

## 风险 4：后台验证结果和机器人真实回复不一致
### 应对
钉钉对话验证接口必须尽量贴近现有回复结构，并在联调阶段重点验证。

## 风险 5：固定报价数据不规范导致误答
### 应对
缺关键字段不可发布，多价格冲突必须走待确认。

---

## 11. 验收标准

### 11.1 主链路验收
至少打通以下 3 条：

1. FAQ 录入 -> 钉钉验证 -> 发布到机器人
2. 固定报价录入 -> 钉钉验证 -> 发布到机器人
3. 制度文档上传 -> 确认 -> 钉钉验证 -> 发布到机器人

### 11.2 权限验收
- 人事不能编辑固定报价
- 商务不能编辑 FAQ / 员工手册
- 财务没有新增/发布权限
- 管理员有所有权限

### 11.3 体验验收
- 非技术同事能够理解后台操作
- 上传、验证、发布路径不绕
- 错误提示清晰可读

## 12. 后续进度记录要求

后续实施进度**统一记录到**：
- [memory-bank/progress.md](memory-bank/progress.md)

### 记录原则
1. 每完成一个明确实施步骤，就更新一条记录。
2. 记录中必须包含：
   - 日期
   - Step ID / 自定义实施标识
   - 状态（`IN_PROGRESS` / `DONE`）
   - 验证证据（测试、联调、校验结果）
   - 变更说明
   - `Skills:` 追溯
3. 本后台实施后续建议使用单独前缀记录，例如：
   - `KB-OPS-01`
   - `KB-OPS-02`
   - `KB-OPS-HF1`

### 建议记录方式
- 方案确定：记 checkpoint
- 主链路打通：记 DONE
- 热修复：记 HF
- 仅做状态同步：记 OPS-CHECKPOINT
