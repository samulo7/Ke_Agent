## 2. 背景与问题定义

当前 20-40 人规模公司的内部协作，存在以下典型问题：

1. 员工遇到制度、流程、资料类问题时，通常需要去钉盘、群聊或直接找人事、财务、商务询问，查找路径分散，耗时高。
2. 人事、财务、商务等岗位长期承担大量重复答疑，例如：
   - 某份制度文件在哪里
   - 某类文档如何申请
   - 出差报销要准备什么材料
   - 请假流程入口在哪里
   - 某个固定配件的标准报价是多少
3. 某些资料具有权限或审批要求，员工不知道是否能直接获取，也不知道应该联系谁、填什么、走什么流程。
4. 现有钉钉审批流程已存在，但员工对规则、入口、所需材料理解不清，导致人工解释成本高、流程反复率高。

因此，需要建设一款接入钉钉机器人的企业内部 Agent，优先通过自然语言对话完成"识别需求、返回答案、给出指引、生成申请内容"，减少员工找信息成本和岗位重复答疑压力。

---

## 3. 产品定位

本产品是一款面向公司内部员工使用的**钉钉单聊机器人 Agent**。

**MVP 阶段定位：**
- 作为员工的内部信息与办事助手
- 作为人事、财务、商务等岗位的答疑分流入口
- 作为制度、流程、文档和 FAQ 的统一查询入口

**MVP 不定位为：**
- 独立审批系统
- 文档管理系统
- 复杂业务系统操作终端
- 替代人类做最终权限审批与敏感资料发放

---

## 4. 产品目标

### 4.1 业务目标

1. 降低人事、财务、商务等岗位的重复答疑量。
2. 提升员工查制度、找资料、问流程、办事情的效率。
3. 将分散在钉盘、人工经验、FAQ 中的信息收敛到统一入口。

### 4.2 MVP 产品目标

1. 支持员工通过钉钉单聊向 Agent 提问。
2. 支持基于内部文档和 FAQ 返回制度、流程、材料要求、固定报价等答案。
3. 支持对有权限要求的资料进行识别，并输出申请方式或申请单草稿。
4. 支持对报销、请假等现有流程提供规则说明与入口指引。
5. 在知识未命中、权限不足或场景超范围时，明确提示并引导到人工或现有流程。

### 4.3 成功指标

上线后按月观察以下指标：

| 指标 | 说明 | 目标 |
|------|------|------|
| 重复答疑量下降 | 人事、财务、商务反馈的高频重复问题数量下降 | 上线 1-2 个月后目标场景下降 **30%** |
| 自助解决率 | 员工无需人工介入即可完成查找或进入流程的比例 | 目标场景达到 **60%** |
| 响应效率 | 员工从提问到拿到有效答案的平均时长 | 常见问题 **10 秒**内返回首条有效结果 |
| 命中满意度 | 通过简单反馈或运营统计衡量 | 正向反馈率达到 **80%** |

> 以上数值为 PRD 默认目标，实际立项时可按公司现状调整。

## 5. 用户角色

### 5.1 主要用户（普通员工）
- 查询制度、流程、资料、所需材料
- 获取办事入口
- 生成申请内容

### 5.2 次级用户

| 角色 | 职责 |
|------|------|
| 人事 | 维护行政、人事、制度类 FAQ 和文档；接收文档申请或转人工问题 |
| 财务 | 维护报销、费用、财务制度类知识；接收复杂财务问题 |
| 商务 | 维护固定价目表或标准报价 FAQ；接收非标准报价咨询 |

### 5.3 管理角色（知识管理员 / 系统管理员）
- 配置知识源
- 管理 FAQ
- 配置敏感资料标签、联系人、转人工规则
- 查看问答日志和命中情况

## 6. 使用场景与用户故事

### 场景 A：制度查询
员工想知道"宴请标准是什么""财务制度在哪里看""请假规则是什么"。

- 作为员工，我希望直接问机器人制度问题，而不是自己翻找钉盘或询问同事。
- 作为管理岗位，我希望制度类问题优先被机器人消化，减少重复解释。

### 场景 B：文档申请
员工需要某份内部文档，但该文档可能涉及权限、审批或人工判断。

- 作为员工，我希望机器人告诉我这类文档是否可申请、找谁、需要填写什么信息。
- 作为人事，我希望机器人先收集完整申请信息，再把标准化申请内容交给我处理。

### 场景 C：报销流程指引
员工需要办理出差报销，但不清楚电子表单、打印、发票粘贴、钉钉审批流等要求。

- 作为员工，我希望机器人一步讲清楚报销流程、所需材料和入口。
- 作为财务，我希望报销前置规则能被标准化回答，减少材料不齐和重复沟通。

### 场景 D：请假流程指引
员工需要请假，但不清楚请假入口、规则和注意事项。

- 作为员工，我希望机器人直接告诉我请假该怎么走，而不是自己找审批入口。

### 场景 E：固定报价 FAQ
销售咨询某类标准配件价格，如"XX 定影器多少钱"。

- 作为销售，我希望机器人能先回答标准报价 FAQ，提高响应速度。
- 作为商务，我希望仅固定价目表内的问题由机器人回答，非标准报价继续由人工处理。

## 7. MVP 范围

### 7.1 In Scope
1. 钉钉单聊机器人接入
2. 用户身份识别
3. 自然语言问题理解与意图分类
4. 基于内部文档和 FAQ 的检索问答（RAG）
5. 基于权限规则的结果过滤与提示
6. 文档申请类请求的申请单草稿生成
7. 报销、请假等固定流程的规则说明与入口指引
8. 固定价目表类 FAQ 查询
9. 未命中后的兜底回复与转人工引导
10. 最小运营能力：问答日志、命中记录、知识维护

### 7.2 Out of Scope
1. 在钉钉内自动代用户发起或提交审批
2. 自动审批或自动发放敏感文档
3. 接入复杂业务系统的实时价格或订单数据
4. 群聊机器人协作场景
5. 多轮复杂业务办理编排
6. 面向外部客户的服务能力

## 8. 解决方案概述

员工通过钉钉单聊向 Agent 提问，系统识别用户身份和问题意图，并按以下逻辑处理：

1. **意图判断**：制度/流程查询、文档申请、报销/请假指引、固定报价 FAQ、无法识别
2. **知识检索**：从 FAQ 或内部文档（RAG）中检索答案
3. **权限校验**：根据用户权限和资料敏感级别决定返回方式
4. **结构化输出**：返回可执行的回复，而不是只返回文档片段

> **技术底座说明**：LLM 采用 **通义千问（Qwen）**，检索层基于 RAG（检索增强生成）能力，知识向量化存储，支持语义检索。

## 9. 功能需求

### FR-01 钉钉单聊接入

系统需支持员工通过钉钉单聊与机器人交互。

**需求说明：**
- 仅支持单聊，不支持群聊（群聊为二期规划）
- 需识别发起人身份，作为权限判断与日志记录基础
- 支持文本消息和**卡片消息（Card）**；卡片消息用于结构化内容展示，如流程指引、申请单草稿
- MVP 不要求支持图片、语音、文件上传解析

**验收标准：**
- 员工可在钉钉中搜索并单聊机器人
- 机器人能接收文本问题并返回文本或卡片结果

---

### FR-02 身份识别与用户上下文

系统需识别当前提问人身份，并关联基础组织信息用于权限判断。

**需求说明：**
- 最少识别：用户唯一身份标识、姓名、部门
- 用户信息用于：内容权限过滤、不同部门 FAQ 差异化展示、日志归属

**验收标准：**
- 同一问题由不同权限用户发起时，系统可输出不同结果或不同提示

---

### FR-03 意图识别

系统需对用户输入进行分类，并路由到对应处理逻辑。

**MVP 意图类型：**

| 意图类型 | 示例输入 |
|----------|----------|
| 制度 / 流程查询 | "宴请标准是什么" |
| 文档申请 | "我要申请 XX 文件" |
| 报销指引 | "出差报销怎么弄" |
| 请假指引 | "我要请假" |
| 固定报价 FAQ | "XX 定影器多少钱" |
| 其他 / 无法识别 | 超出范围的问题 |

**验收标准：**
- 对典型问题能正确识别到上述类别
- 无法识别时不编造答案，进入兜底流程

---

### FR-04 文档与 FAQ 检索问答

系统需支持基于内部文档与 FAQ 的统一查询能力（RAG 检索）。

**需求说明：**
- 首版知识底座以内部文档检索为主，FAQ 作为补充
- 返回内容应包含：简洁答案、适用条件或说明、来源文档名称或 FAQ 标识、原文入口（如可用）
- 检索到多个候选结果时，优先返回最相关结果，可附带"你是否想找"列表

**验收标准：**
- 对制度、流程、材料要求类问题可返回明确答案
- 返回内容包含来源信息，不做黑盒回答

---

### FR-05 权限控制

系统需沿用现有权限原则，不扩大资料访问范围。

**规则要求：**
1. 用户原本有权限访问的资料，Agent 可返回内容摘要与原文入口
2. 用户无权限或资料为敏感资料时，Agent 不返回正文
3. 对受控资料，Agent 需返回：不可直接查看的提示 + 申请方式/联系人/审批路径 + 申请单草稿入口（如适用）

**验收标准：**
- 无权限用户查询敏感资料时，系统不泄露正文内容
- 有权限用户查询同一资料时，系统可正常返回可见内容

---

### FR-06 文档申请单草稿生成

对于"我要申请 XX 文档/材料"的场景，系统需生成标准化申请内容。

**申请单草稿至少包含：**
- 申请人姓名
- 所属部门
- 申请资料名称
- 申请用途
- 期望使用时间
- 建议审批对象或处理岗位

**交互要求：**
- 信息不完整时追问必要字段，**追问最多 2 轮**
- 超时（用户超过 5 分钟无回复）或用户中断后，输出兜底提示，引导用户直接联系对应岗位
- 用户确认后，输出可复制、可转发的申请内容（卡片格式）
- MVP 不要求自动提交审批

**验收标准：**
- 对文档申请场景，系统能生成结构化申请文本
- 生成内容字段完整，可直接给人事或领导审核

---

### FR-07 流程规则说明与入口指引

系统需对已有公司流程提供标准化说明和入口指引。

**首批支持场景：** 出差报销、请假

**返回内容至少包括：** 办理条件、所需材料、操作步骤、常见错误、钉钉入口或责任岗位

**验收标准：**
- 员工提问"出差报销怎么弄""我要请假"时，可在一轮内获得完整指引

---

### FR-08 固定报价 FAQ

系统需支持固定价目表范围内的报价问答。

**规则要求：**
- 仅回答已维护在 FAQ 或固定价目表中的标准项
- 返回报价时需带上适用条件、版本日期或备注
- 超出固定范围时，明确提示"需联系商务确认"

**验收标准：**
- 对标准品报价可返回准确答案
- 对非标准项不编造价格

---

### FR-09 兜底与转人工

系统需在知识未命中、权限受限或问题超范围时，给出明确兜底方案。

**兜底类型及处理：**

| 兜底类型 | 回复内容要求 |
|----------|-------------|
| 未找到相关资料 | 说明未找到原因 + 建议联系人或部门 |
| 找到资料但无权限 | 说明不可查看 + 申请路径 |
| 场景超出 MVP 范围 | 说明暂不支持 + 引导人工 |
| 问题过于模糊 | 追问补充信息（最多 1 轮） |

**验收标准：**
- 任意异常场景下，系统不返回空白结果或编造答案

---

### FR-10 知识维护与运营基础能力

**MVP 最低要求：**
- FAQ 维护入口
- 文档知识源更新机制（更新后 **30 分钟内**生效）
- 问题日志记录
- 未命中问题统计
- 高频问题统计

**验收标准：**
- 管理员可查看哪些问题命中、未命中和最常被问

## 10. 关键流程

### 10.1 制度查询流程
```
员工输入问题 → 意图识别（制度查询）→ RAG 检索 → 权限校验 → 返回摘要/来源/入口
```

### 10.2 文档申请流程
```
员工输入申请需求 → 意图识别（文档申请）→ 判断是否受控 → 追问缺失字段（≤2轮）
→ 生成申请单草稿（卡片格式）→ 提示提交给指定岗位
```

### 10.3 报销 / 请假指引流程
```
员工输入 → 意图识别 → 返回规则说明/材料/步骤/入口 → 如追问，继续从知识中回答
```

### 10.4 报价 FAQ 流程
```
员工输入 → 意图识别（报价 FAQ）→ 价目表查找 → 命中：返回价格+版本日期 / 未命中：提示联系商务
```

## 11. 知识源与内容策略

### 11.1 首版知识源
1. 公司内部制度文档（行政、财务、人事）
2. 流程说明文档（请假、报销、材料申请）
3. 固定价目表或标准报价 FAQ
4. 人事、财务、商务整理的高频 FAQ

### 11.2 内容要求
- 知识需标记**来源、更新时间、责任人**
- 敏感资料需标记权限级别和处理方式
- 报价类知识需标记**版本日期**，避免过期回答

---

## 12. 权限与安全要求

### 12.1 权限原则
- 沿用现有权限，不通过 Agent 放大权限
- 优先控制"可答复什么"和"不可答复时如何引导"

### 12.2 安全要求
1. 对敏感资料不返回完整正文
2. 日志中保留问答记录，用于审计和优化
3. 答案涉及来源文档时，可追溯到知识源
4. 管理员能力需区分普通用户权限

### 12.3 风险控制
- 避免 LLM 编造答案（通过 RAG 召回 + prompt 约束）
- 避免过期报价造成业务误导（强制展示版本日期）
- 避免无权限泄露内部资料

---

## 13. 非功能需求

| 维度 | 要求 |
|------|------|
| 响应性能 | 常见问题首条回复时间目标：**10 秒**内；超时需返回"处理中/请稍后重试" |
| 可用性 | 回复结构简洁，优先给结论，再给步骤和来源；不要求用户理解复杂术语 |
| 可维护性 | FAQ 可按角色独立维护；知识更新后 **30 分钟**内生效 |
| 可审计性 | 支持追踪问题、答案、来源、用户、时间 |

## 16. 风险与依赖

### 16.1 主要风险

| 风险 | 对应策略 |
|------|----------|
| 知识源不完整，命中率低 | 首版聚焦高频问题，不追求大而全 |
| 权限规则不清，内容泄露或体验过严 | 对敏感资料优先保守处理，只给申请路径 |
| 报价 FAQ 更新不及时，错误回答 | 报价类答案强制展示版本时间 |
| 申请流程未标准化，申请单生成后仍需大量人工确认 | 对未命中问题建立补录机制，逐步迭代 |
| Qwen API 可用性或延迟影响体验 | 设置超时兜底回复，监控 API 响应时间 |

## 17. 验收标准

### 17.1 制度查询
- 输入"宴请标准是什么"
- 预期：返回制度摘要 + 适用范围 + 来源文档名称 + 原文入口（有权限时）

### 17.2 文档申请
- 输入"我要申请 XX 文件"
- 预期：识别为文档申请 → 追问用途等字段（≤2轮）→ 生成结构化申请单草稿 → 不直接返回敏感文件正文

### 17.3 出差报销
- 输入"出差报销怎么弄"
- 预期：返回流程步骤 + 所需材料 + 钉钉办理入口 + 常见错误提示

### 17.4 请假
- 输入"我要请假"
- 预期：返回请假规则 + 入口 + 必要注意事项

### 17.5 固定报价
- 输入"XX 定影器多少钱"
- 预期：标准价目表内 → 返回价格 + 适用说明 + 版本日期；不在范围 → 提示联系商务

### 17.6 无权限查询
- 输入某份受控资料名称
- 预期：不返回正文 + 提示不可直接查看 + 返回申请路径/联系人

### 17.7 未命中问题
- 输入系统未覆盖的问题
- 预期：明确说明未找到答案 + 不编造内容 + 提供联系人或后续处理建议

### 二期重点场景 A：请假智能辅助

**场景目标：** 在不替代钉钉请假审批流的前提下，使用 AI 优化员工在请假前、中、后的操作体验，减少假别选错、材料遗漏和审批退回。

**能力边界：**
- AI 负责：规则解释、假别推荐、材料提醒、申请内容生成、流程提醒
- 钉钉继续负责：假期申请提交、余额管理、审批流转和最终审批
- AI 不替代审批人判断，不绕开现有请假制度

**功能需求：**
1. **假别推荐**：根据自然语言描述推荐假别并说明理由，明确提示"最终以员工提交和公司制度为准"
2. **规则解释**：回答天数、适用条件、是否需要证明材料、审批要求等，引用制度来源
3. **材料提醒**：根据假别自动提示所需材料（如病假证明、产检材料等）
4. **请假单内容生成**：整理结构化请假信息，适合复制到钉钉请假单或审批备注
5. **余额与合规预判**：如接入钉钉假期余额，提示余额是否充足；未接入时提示"请以系统实际余额为准"
6. **风险提醒**：对假别选择不合理、附件缺失、连续天数超阈值等情况给出提示
7. **提交后提醒**：请假通过后，提醒补充交接信息、确认返岗、补交材料

**典型流程：**
```
员工输入"我明天想请假去医院复查"
→ AI 追问时间、预计天数、是否已有证明
→ AI 推荐假别并说明依据
→ AI 提示所需材料和风险点
→ AI 生成可提交的请假说明文本
→ AI 引导用户跳转到钉钉请假流程完成正式提交
```

### 二期重点场景 B：招聘筛选辅助

**产品形态：** 招聘网站页面 + 浏览器插件/侧边栏 + 内部 Agent 服务（钉钉作为结果通知和协作分发渠道）

**能力边界：**
- AI 负责：简历解析、岗位匹配、筛选理由生成、风险提示、追问建议
- HR 继续负责：最终筛选、邀约、淘汰和录用判断
- AI 不基于年龄、性别、婚育等敏感信息做自动筛选结论
- AI 不默认支持批量抓取整站简历

**功能需求：**
1. **简历解析**：HR 打开候选人详情页后，触发 AI 解析当前页面内容，输出结构化候选人卡片
2. **JD 匹配分析**：输出匹配点、不匹配点、缺口和综合建议；结论采用"推荐沟通/可进一步评估/不优先"分层
3. **筛选理由生成**：给出可解释的判断依据，如行业经验匹配、岗位层级匹配、稳定性一般等
4. **风险提示**：标记频繁跳槽、岗位跨度大、关键信息缺失等需重点核实项
5. **首轮沟通问题建议**：根据简历和 JD 自动生成追问问题
6. **候选人摘要生成**：将长简历压缩为 5-8 行摘要
7. **结果沉淀**：支持将 AI 分析结论同步到内部人才库、钉钉消息或内部表格

## 19. 默认假设

1. 公司已使用钉钉，并存在基础审批流程。
2. 公司允许员工通过单聊机器人获取内部制度、流程和 FAQ 类信息。
3. 对敏感资料，默认"保守返回"，优先提示申请路径，而不是直接给内容。
4. 首版不要求完全自动化，更重视可控、可上线、可运营。
5. LLM 采用通义千问（Qwen），通过 RAG 检索内部文档作为知识底座。
6. 技术部署方式在本 PRD 中不锁定具体架构，只约束权限、安全和可追溯要求。
